# Project Context Records (PCR)

[English](./README.md) · **简体中文**

> 一套面向 AI 时代的方法论：在仓库里维护一份持久的、随代码版本化的项目**元上下文（meta-context）**
> 档案——*为什么*、架构、维护者的决策、意图——让一个会失忆、又自作主张的 AI 协作者**继承**
> 项目的判断，而不是从零重新推导。

PCR 是 **context engineering（上下文工程）** 的一个实践。Context engineering 经营"喂进模型上下文里
的东西"；PCR 是其中**项目范围、且落盘持久**的那一份——值得版本化、值得 review、值得跨会话继承的
那部分。

## TL;DR：如何使用

把下面这段粘进你项目的 `AGENTS.md` / `CLAUDE.md`（把它指向你存放记录的目录）：

```markdown
## Project Context Records (PCR)

This project follows **Project Context Records (PCR)** —
methodology: https://github.com/hyf0/project-context-records. PCR keeps the
project's durable design context — the *why*, the decisions, the architecture —
so you inherit it instead of re-deriving or re-litigating what's already settled.

When working here:
- **Where they live.** Records are in `.agents/docs/` — one concept per file,
  cross-linked with relative Markdown links (`[name](./name.md)`).
- **Read first.** If a record covers the area you're touching, read it before acting.
- **Keep it fresh.** If your change affects a record, update it in the same change —
  a stale record is a trap, not an asset.
- **Provenance.** An unstamped line is AI-accumulated: challenge and verify it freely.
  A line stamped `[✓ @handle]` was vouched for by a human — don't reopen or
  re-verify it. Only a human adds a stamp.
```

这段就是采纳的全部。它把每个协作者——无论人还是 agent——路由到记录，并写明他们要遵守的规则。
它**故意做薄**：agent 每个会话会自动读 `AGENTS.md`，但**不会**主动 fetch URL，所以只把"每次会话
必须服从"的内联进去；其余的留在一个链接之外，agent 无需 fetch 就能正确行动。

然后就只管干活：一边工作一边**自由积累（accumulate）**记录，并在有人 review 时
**蒸馏（distill）**它们——把有价值的提升、把过期的剪枝、给人背书的盖章。

**下面全部是 *why*。**

## 前提：输出受限于上下文

LLM 的输出是其上下文的函数。给它决定答案的那部分上下文，结果就朝项目想要的方向走；饿着它，它就
用自信的猜测填补空缺——而错而似是而非的活，正是这么来的。

不对称才是关键。上下文多有代价（噪音、注意力被稀释、token）；而*缺了决定答案的那条上下文*则毫无
代价可言——它就是纯输。一个有能力的模型，在缺少决定性信息时照样产出错误的活；它的天花板由它能
看到的东西决定。

这就是为什么 PCR 有两半：**积累（accumulate）**那些本会蒸发的判断（太少一定坏），以及
**蒸馏（distill）**它们——剪掉噪音、让当前真相保持新鲜（太多或过期也坏）。

## 为什么代码不再够用

代码记录的是**结果态**：系统最终长成什么样。它在结构上丢掉了协作者需要的三样东西：

1. **被否决的路径。** 代码只显示"我们选了 A"，从不显示"我们考虑过 B 和 C；B 因为 X 被毙了"。
   于是下一个协作者看到 A，问"为什么不用 B？"，真去试 B，撞回同一面墙。
2. **一个选择的状态。** 这是深思熟虑、已关闭的决策，还是待 review 的权宜之计？代码两种情况长得
   一模一样，但二者要求的行动相反。
3. **一个判断的出处。** 它建立在什么约束、什么权衡、谁的拍板之上？约束变了，决策可能就该重开——
   但前提是你记下了它当初挂在什么上面。

在前 AI 时代，这些活在维护者的脑子里，靠**连续性**传递：code review、师徒带教、"去问写它的人"。
团队是连续的；知识是隐性的，但不断裂。

AI 协作者**没有连续性**。每个会话都是一个失忆的、有能力的、又*自作主张*的新人。它不会"去问写它
的人"——它自己推理、自己怀疑、自己动手。隐性知识对它等于不存在。所以那一直存在、却藏在人脑里的
判断层，如今必须被做成**显式的制品**——否则它每个会话蒸发一次，永远如此。

PCR 就是那个制品。它保存的不是*代码是什么*，而是*判断是怎么来的*——让已经付过的思考不必再付
一次。

## 老树，新芽

PCR 是 **Architecture Decision Records（ADR）** 的 AI 时代后裔——ADR 这套约定由 Michael Nygard 于
2011 年提出，后来长成了一个[工具与模板的生态][adr-hub]。一条 ADR 把每个重要决策记成一个小的、带
编号、随版本管理的文件，写明它的*上下文（context）*、*决策（decision）*与*后果（consequences）*；一条
已定的记录从不修改——要改它，你写一条新的去**取代（supersede）**它，留下一条只增不改的痕迹。

PCR 保留了 ADR 最好的那个本能——*知识活在仓库里，跟代码一起 review、一起版本化，而不是塞进
wiki*——并为新读者嫁接了三根新芽：

1. **决策 → 上下文。** ADR 记的是一次性的选择。PCR 记的是整个元层：决策*以及*架构、心智模型、
   刻意的偏离、坑、以及在途的计划。一次性决策只是项目上下文的一个子集，不是全部。
2. **不可变且堆叠 → 单一且新鲜。** ADR 靠冻结每个已定文件、再追加一个取代它的文件来保存历史；
   当前真相最终散落在一条 supersede 链上。PCR 把**当前结论放在一处、保持新鲜**，把演化交给
   **git**。一个失忆的 agent 为了找今天的答案去重读整条 supersede 链，纯属浪费；给它一个可信的
   答案，把历史交给 log。
3. **读者＝人 → 读者＝自作主张的失忆 agent。** ADR 只描述，因为人读完后果会自我约束。而 agent
   会"发现"一件已定的事，然后动手。所以一条 PCR 记录可以带一个**给 agent 的行动指令**（"这个不在
   契约内；别再标记它"），而不只是描述。

## 记录什么

一个项目的**元层（meta-layer）**：它为什么长这样、架构、维护者决策、意图、刻意的偏离、心智模型、
来之不易的坑、以及当下在途的计划。

往往更锋利的是它的负向定义：**任何关于这个项目为真、却在代码里看不见的东西。**

捕捉是**宽的**——任何有价值的，都随手、廉价地记下来。价值在之后由**蒸馏**（见下）来筛，不在捕捉时
筛。短命的东西（一份实现计划、在途的推理）可以临时追踪；熬过蒸馏的，才是有价值的残留。

## 机制：provenance 盖章

这是唯一的硬通货，它立在一条第一性原理上：

> 在 AI 时代，文本是廉价的、且**默认由 AI 撰写**。稀缺而值钱的动作，是**一个人在某个论断上盖下
> 他的章。**

所以规则只有一个 bit：

- **无章 = AI 积累的。** 默认底料。当作可质疑的——质疑它，随便去验证它。
- **`[✓ @handle]` = 一个人 review 过、并为它背书。** 别重开它，别浪费轮次去重验它，把它当地基。

```
时间戳一律以 UTC 存储，只在边缘转换 —— 一个 DST bug 之后定下。   [✓ @hyf0]

把 cache TTL 提到 1 小时大概是安全的。   ← 无章：AI 积累的，随便验证
```

**人的参与是可选的——一个质量增益，不是构成性的部分。** AI 积累的底料自己就能转；人的盖章只是
*提升*选中的条目。你把人稀缺的注意力花在赌注值得的地方，其余的留作诚实的默认。

**为什么只要一个 bit。** 一个 AI 知识库最深的失败模式，是 AI 积累的论断**冒充人背书的法律**——
后来的 agent 把机器自己的猜测当成既定教条。一个 bit 恰好治这个。其余的一切——*这是决策还是观测？
已定还是暂行？*——都写进条目的**散文**里，而不是塞进更多的章。从一个 bit 起步；让真实使用去拉出
更多，如果它真的需要的话。别预先把柜子塞满。

## 生命周期：积累 → 蒸馏

- **积累（accumulate）**（廉价、默认、由 agent 做）：一边工作一边持续记录上下文——包括短命的计划
  和在途的推理。量大没关系；有噪音是预期之内的。
- **蒸馏（distill）**（稀缺、由人做）：一次 review，把有价值的**提升**、把错的和过期的**剪枝**、给
  reviewer 背书的**盖章**。蒸馏是廉价积累变成持久、可信记录的途径——它是被当作一项常规实践来施加
  的人类盖章，也正是"临时追踪的实现计划"变成"留下来的那个有价值的东西"的地方。

## 写法约定

- **一概念一文件。** 用标准相对 Markdown 链接（`[other-concept](./other-concept.md)`）在相关条目间
  互链——GitHub 上可点、无歧义、工具无关。
- **保持当前真相新鲜。** 漂移是致命风险：一条不再符合现实、却自信的记录，会从资产翻成**陷阱**——
  agent 会满怀信心地照它行动。一条过期的记录比没有记录更糟。当你的改动影响到某条记录时，在同一次
  改动里更新它。
- **记下 why**，不只是 what：权衡、被否的备选、已知的坑。以原理或直觉开头。
- **写到让协作者不读代码内部就能 get。** 条目作为散文自立；agent 读它，就*懂了*。

## 刻意搁置

下面这些在设计时被考虑过，又**刻意从 v1 砍掉**，为的是把机制保持在一个 bit。记在这里，是为了让
"否决"可追溯，而不是被悄悄遗忘（这一节本身就是 PCR 在践行自己）：

- **认知状态子章**（`observed` / `inferred` / `decided` / `open`）：更丰富，但用四个章去做一个 bit
  已经扛起的那件真正要紧的事。
- **`settled` vs `provisional`**（决策的死活）：真实的区分，但眼下写在条目的散文里就够了。
- **新鲜度 / 漂移钩子**（例如 `@baseline ink-v7.0.4`，基线一移就自曝过期）：有前景，但在使用真正
  提出需求之前，徒增仪式。
- **每条记录的"何时该读我"触发条件**（类似 skill 的 `description` 路由）：规模化时有价值，v1 阶段
  过早。

以上全部的指导原则：**从简单起步；让真实使用去拉出更多。** 任何对这种极简主义的偏离，本身都应是
一个盖了章、经人背书的决策。
